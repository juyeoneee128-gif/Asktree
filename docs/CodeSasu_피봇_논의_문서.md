## 1. 왜 피봇이 필요한가

### 현재 문제: 메인 기능의 정확도가 낮습니다

현황 탭(구현률)과 이슈 탭(코드 감지)이 Asktree의 핵심 기능인데, 테스트 결과 정확도가 기대에 미치지 못합니다.

- **현황 탭**: 구현률이 100%로 표시되지만, 실제로는 세부 구현 항목이 "0/8"로 비어있으면서 "구현 완료"로 판정됨
- **이슈 탭**: 동일한 코드를 분석해도 매번 결과가 달라짐 (이슈 그룹핑 불완전, 감지 건수 변동)
- **현황 탭 갱신**: "다시 분석" 버튼을 매번 눌러야 하고, 새로고침과 혼동됨

### 근본 원인 진단: 1~2단계(수집/정돈)가 허술합니다

Asktree의 데이터 파이프라인을 4단계로 분석했을 때:

| 단계 | 현재 상태 | 문제 |
|------|-----------|------|
| 1. 수집 | 에이전트가 raw JSONL + git diff를 수집 | 수집 자체는 되지만 파싱 없이 원본 그대로 저장 |
| 2. 정돈 | 없음 | raw 데이터를 의미 있는 단위로 구조화하는 과정이 없음 |
| 3. 시각화 | 현황 탭 구현률, 이슈 탭 | 정돈 안 된 데이터 위에 시각화하니 정확도 저하 |
| 4. 감지 | 이슈 탭 | 기반 데이터가 부실하면 감지도 부실 |

**결론**: 3~4단계(시각화/감지)의 정확도를 올리려면 1~2단계(수집/정돈)를 먼저 강화해야 합니다. 현재 상태로 배포하면 유저가 부정확한 데이터를 보게 되어 신뢰를 잃을 위험이 있습니다.

## 2. Memory.inc 레퍼런스 차용 배경

### Memory.inc란?

프라이머, 더벤처스 투자를 받은 AI 기억 레이어 서비스입니다. 흩어진 데이터(Slack, Notion, GitHub 등)를 한 곳에 모아 관계형으로 맥락을 관리하고, AI가 필요한 맥락만 정확히 전달받을 수 있게 합니다.

핵심 구조: **찾기 → 답하기 → 저장하기 → 이어쓰기**

### Asktree(→ CodeSasu)와의 공통 근본 문제

"AI와 작업할 때 맥락 유실과 과정 트래킹이 불가능하다"

- Memory.inc: 여러 서비스에 흩어진 데이터의 맥락 유실 해결
- CodeSasu: 코딩 과정에서 발생하는 코드 변경/유실의 맥락 트래킹 해결

### 구체적으로 참고하는 부분

Memory.inc의 **"데이터 수집 → 효과적 아카이빙"** 베이스 레이어를 참고합니다.

| Memory.inc | CodeSasu 적용 |
|------------|-------------|
| 흩어진 데이터를 한 곳에 모음 | 세션 로그를 파싱하여 "이 세션에서 한 일" 자동 요약 |
| 출처와 함께 답함 | 이슈 감지 시 "어떤 세션에서, 어떤 변경으로 인해" 출처 표시 |
| 반복될 내용은 메모리로 남김 | CLAUDE.md 보호 규칙 누적 (기존 기능 유지) |
| 다음 사람과 다음 AI가 이어받음 | 세션 요약을 CLAUDE.md에 자동 반영 → 다음 세션 맥락 연결 (Phase 2) |

## 3. 고도화 계획

### 3-1. 에이전트 자동 수집 확장

**변경 전**: 유저가 웹에서 기획서를 수동 업로드
**변경 후**: 에이전트가 push할 때 프로젝트 내 `docs/*.md` 파일을 자동 수집

- 코드 변경 (git diff) + 세션 로그 (JSONL) + 기획 문서 (docs/*.md)를 한 번에 수집
- 유저는 에이전트 설치 1회만으로 모든 데이터가 자동 수집됨
- 기획서가 업데이트되면 다음 push 때 자동 반영 → 현황 탭도 자동 갱신

### 3-2. 세션 요약 파싱 (정돈 레이어 강화)

**변경 전**: 세션 탭에 raw JSONL만 표시 (유저가 읽을 수 없음)
**변경 후**: JSONL을 파싱하여 구조화된 세션 요약 자동 생성

MVP에서 구현하는 것 (규칙 기반 파싱, API 호출 불필요):
- 유저 프롬프트 목록 추출
- 변경된 파일 목록 + 작업 내용
- 도구 사용 횟수 (Write, Bash 등)
- 세션 시작/종료 시간
- 에러 발생 여부

Phase 2에서 추가하는 것 (AI 기반, 크레딧 차감):
- 자연어 세션 요약 ("이 세션에서: 결제 모듈 추가, 인증 미들웨어 수정")
- 세션 간 맥락 연결
- CLAUDE.md 자동 반영 (이어쓰기)

### 3-3. push 시 자동 분석 (UX 개선)

**변경 전**: 유저가 웹에서 "다시 분석" 버튼을 매번 눌러야 함
**변경 후**: push 시 이슈 분석 + assess 자동 실행 → 유저가 웹에 올 때 이미 최신 결과

- "수동 재분석" 버튼은 기획서 교체 후 재분석 용도로만 유지
- 탭 조회는 이미 분석된 결과를 표시하는 것이므로 무료

## 4. 서비스명 변경: Asktree → CodeSasu (코드사수)

### 변경 배경

인사이트 분석을 통해 서비스 정체성을 재정립했습니다.

- **인사이트 1**: 국내에서 바이브코딩으로 제품을 만들려는 1인 빌더보다, 실무 업무 자동화에 AI 코딩을 쓰는 현직자가 훨씬 많음. 다만 두 그룹 모두 "코드를 볼 줄 모르는 비개발자로서 코딩 과정을 통제할 수 없다"는 동일한 근본 문제를 겪고 있음
- **인사이트 2**: Memory.inc 레퍼런스를 통해 "데이터 수집 + 정돈"이 기반 레이어로서 중요하다는 것을 확인. "사수 개발자처럼 옆에서 과정을 랩업/팔로잉하고 문제가 있으면 먼저 알려주는 솔루션"이라는 정체성 도출

### CodeSasu (코드사수) 이름의 의미

중의적 표현:
1. **코드 사수 (先輩)**: 옆에서 사수처럼 도와주는 존재
2. **코드를 사수하다 (死守)**: 프로젝트 코드를 지키고 보호한다

### 브랜딩

| 항목 | 내용 |
|------|------|
| 서비스명 | CodeSasu (코드사수) |
| 메인 카피 | "내 손안의 사수 개발자" |
| 서브 카피 | "내 코딩 과정을 기억하는 사수: 코드를 몰라도 괜찮아요. 사수가 지켜보고 있으니까." |
| SEO 충돌 | 없음 (크로스체크 완료) |

## 5. 크레딧 정책 변경

### 변경 전
- 모든 작업 1 크레딧 고정
- 가입 시 10 크레딧 (5~10회 분석)
- 현황 조회에도 "다시 분석" 필요 → 크레딧 차감 → 유저 거부감

### 변경 후
- 토큰 기반 크레딧 (1 크레딧 = 5,000 토큰)
- 가입 시 500 크레딧 (약 25~30회 push 분석, 2~3주 분량)
- Claude API 호출에만 크레딧 차감, 결과 조회는 무료
- push 시 자동 분석이므로 유저가 크레딧을 의식하지 않고 사용

| 행위 | 크레딧 |
|------|--------|
| push 시 자동 이슈 분석 | 차감 (약 8~12) |
| push 시 자동 assess | 차감 (약 4~8) |
| push 시 기획서 기능 추출 (docs 변경 시) | 차감 (약 3~5) |
| 탭 조회 (현황/이슈/세션/기획서) | 무료 |
| "수동 재분석" 버튼 | 차감 |
| 보호 규칙 생성/복사 | 무료 |
| Fix 복사 | 무료 |

**설계 원칙**: GitHub처럼, 핵심 기능은 무료로 사용하되 서버 비용이 드는 행위(AI API 호출)에만 크레딧을 차감. 유저가 2~3주 동안 충분히 가치를 체감한 후에 리밋에 도달하도록 설계.

## 6. 앞으로의 실행 순서

| 순서 | 작업 | 설명 |
|------|------|------|
| 1 | 랜딩 페이지 개발 (사전 등록 버전) | 피봇된 정체성/메시지 반영 + 이메일 수집 폼. "CodeSasu — 내 손안의 사수 개발자" |
| 2 | 배포 (Vercel) | 사전 등록 페이지 먼저 라이브 → 수요 검증 |
| 3 | 마케팅 진행 | Reddit, X, 멘토링 다음 기수 등에서 사전 등록 유도 |
| 4 | 에이전트 docs 수집 + 백엔드 자동화 | push 시 docs/*.md 자동 수집 + 자동 분석 + 자동 assess |
| 5 | 세션 요약 파싱 | JSONL → 구조화된 세션 요약 (규칙 기반) |
| 6 | 크레딧 정책 구현 | 토큰 기반 크레딧 차감 + 500 크레딧 무료 제공 |
| 7 | 서비스명 전역 치환 | Asktree → CodeSasu 코드/문서/디자인 전부 변경 (약 1시간) |
| 8 | Beta 오픈 | 사전 등록자에게 Beta 초대 발송 |

**사전 등록 페이지를 먼저 배포하는 이유**: Memory.inc도 이 방식으로 사전 등록 80명을 먼저 모은 뒤 제품을 개발하고 있음. 수요를 검증하면서 동시에 제품을 완성하는 전략.
