# Asktree

**Claude Code 프로젝트의 코드 파손 감지 + 복구 + 보호 도구**

Product Requirements Document v7.0 | 2026.03.20

> "네가 묻지 않아도, AI가 망가뜨린 걸 자동으로 찾아내고, 고치는 법을 알려주고, 다시는 안 망가지게 지켜준다."

---

## 1. 제품 개요

### 1.1 한 줄 정의

**Claude Code로 코딩할 때, AI가 기존 기능을 망가뜨리면 자동으로 감지하고 복구 명령어를 주고, CLAUDE.md에 AI 가이드라인을 추가해서 재발을 방지하는 도구.**

### 1.2 와우포인트

**"감지 → Fix → 보호. 한 번 당한 문제는 두 번 없다."**

### 1.3 3단계 가치 체인

| 단계 | Asktree가 하는 일 | 유저 행동 | 해소하는 격차 |
|------|-------------------|-----------|---------------|
| **1. 자동 감지** | 세션 간 코드 비교로 삭제/변경된 기능을 자동 발견 | Asktree를 열면 이미 결과가 있음. 아무것도 안 해도 됨. | 인지 격차 (뭐가 망가졌는지 모름) |
| **2. Fix 제공** | 복구를 위한 Fact 기반 자연어 CLI 명령어 자동 생성 | Fix를 복사해서 Claude Code에 붙여넣기 | 해결책 격차 (뭘 해야 하는지 모름) |
| **3. CLAUDE.md 가이드라인** | 보호해야 할 기능을 CLAUDE.md AI 가이드라인으로 자동 생성 | '전체 복사' 텍스트를 CLAUDE.md에 붙여넣기 | 재발 방지 (같은 문제 반복) |

### 1.4 핵심 가치

| 가치 | 설명 |
|------|------|
| 자동 감지 | 유저가 묻지 않아도 코드 파손을 먼저 발견. LLM에게 '뭘 물어봐야 하는지조차 모르는' 문제를 해결. |
| Fix 가이드 | 코드가 아닌 Fact 기반 자연어 CLI 명령어. '이전에 이랬는데 지금 이렇게 바뀌었으니 되돌려줘' 형태. 비개발자가 Claude Code에 복사-붙여넣기. |
| CLAUDE.md 가이드라인 | 감지된 핵심 기능을 CLAUDE.md 규칙으로 자동 생성. AI가 함부로 삭제 못 하게 방지. 효과: 30~40% 방어. 핵심은 감지 자체이며 가이드라인은 덤으로 예방. |
| 제로 마찰 | 코딩하면 자동 수집·분석. 유저는 기록/정리/관리 불필요. |
| Privacy-First | 코드 원본은 분석 후 즉시 파기(Ephemeral Processing). |

### 1.5 정체성 원칙

- Asktree는 '보이지 않는 동료'. 바이브코더가 인지조차 못하는 코드 파손을 대신 찾아준다.
- 핵심은 '감지 → Fix → 보호'의 3단계 루프. 시각화나 학습은 목적이 아님.
- Claude Code 전용 (MVP). Cursor/Aider는 Phase 2 확장.
- LLM과의 차별점: LLM은 '물어봐야 답함'. Asktree는 '묻지 않아도 먼저 알려줌' + 세션 간 변경 추적.
- '더 좋은 코드를 만들게 해주는 도구'가 아니라 '만든 것을 잃지 않게 해주는 도구'. 프로젝트의 생존 기간을 늘린다.

## 2. 문제 정의

### 2.1 리서치 기반 핵심 문제

| 문제 | 발생 빈도 | 구체적 현상 |
|------|-----------|-------------|
| **컨텍스트 유실 → 기존 기능 삭제** | 68% | '대시보드 리팩토링해줘' → 3일간 만든 권한 체크+데이터 로직 전부 삭제. 85줄→12줄. |
| **한 파일 수정 → 연쇄 파괴** | 20% | 파일명 변경 → 15개 파일 import 깨짐. DB 컬럼 추가 → 8개 쿼리 에러. |
| AI가 규칙 무시 | 빈번 | CLAUDE.md에 '이 파일 건드리지 마' 써도 긴 대화 후 무시. |
| 문제 인지 자체 불가 | 80% | 코드를 읽을 수 없어서 뭐가 삭제/변경됐는지 모름. |
| **겉으로 멀쩡한 가짜 코드** | 빈번 | AI가 하드코딩된 OK 값만 돌려주는 함수 생성. 에러 로그 없이 잘못된 결과. |
| 데이터 스키마 불일치 | 빈번 | 필수 필드 누락, 예상 타입과 다른 값으로 런타임 NPE 발생. |

### 2.2 타겟 유저의 실제 행동 패턴

- 설계부터 구현까지 AI에게 맡기고, 코드를 읽지 않고 결과(화면) 중심으로만 확인
- 에러가 나면 로그를 복붙해서 AI에게 '고쳐줘'만 반복 → AI death loop
- 컨텍스트 관리를 안 해서 AI가 이전 결정을 잊고 기존 기능을 덮어쓰거나 삭제
- CLAUDE.md의 존재를 모르거나, 알아도 '뭘 써야 할지 몰라서' 사용 못 함 (80%)
- 3~4주 이상 진행하면 AI가 모순된 패치를 내놓아 전체 안정성이 무너짐

### 2.3 왜 LLM만으로는 해결이 안 되는가

- LLM에게 물어보려면 '뭘 물어봐야 하는지' 알아야 함. 바이브코더 80%는 문제 인지 자체가 불가.
- LLM은 '현재 상태'만 봄. '3일 전에 있던 함수가 오늘 사라졌다'는 추적 불가.
- 설계를 아무리 잘 해도 문제는 발생. 원인이 '설계 부족'이 아니라 'LLM의 구조적 한계(Context Rot)'. 설계는 빈도를 줄이고, Asktree는 발생한 것을 잡아낸다.

## 3. 타겟 유저

**사이드프로젝트·1인 SaaS 창업형 바이브코더 (Claude Code 사용자)**

- 직군: 기획자, 디자이너, 크리에이터, 직장인 N잡러
- 목적: 유료 서비스, 니치 SaaS 런칭. 실제 사용자에게 돈을 받는 서비스.
- 특성: 개발은 모르지만 비즈니스적 생존을 위해 코딩을 도구로 선택.
- 타겟 프로젝트 규모: 파일 50개 이하, 기능 10개 이하.
- 1차 타겟: 멘토링 다음 기수 수강생 (실제 유통 채널 확보).

## 4. 솔루션

### 4.1 전체 레이아웃

**프로젝트 홈:** 좌측 사이드바(모든 프로젝트/즐겨찾기/휴지통/관리자 설정/플랜 및 결제) + 프로젝트 카드 그리드. 프로필 드롭다운에 테마/언어/가이드/로그아웃.

**프로젝트 내부:** Manyfast 스타일 좌측 사이드바(현황/검토/CLAUDE.md/Activity/기획서/설정) + 우측 메인 콘텐츠. 프로젝트 드롭다운 최상단에 '← 모든 프로젝트'로 홈 복귀 경로.

**모든 탭 Master-Detail 2-Pane 레이아웃으로 통일** (학습 비용 제로). 빈 상태는 전체 너비 중앙 정렬.

### 4.2 [현황] — 기획 대비 구현 상태

기본 진입 탭. 사이드바에서 [현황] 선택 시 표시.

- **글로벌 헤더:** '기획 대비 구현 현황' + 진행률 % + 상태 배지 (✅구현/🔨부분/🔲미구현/⚠️확인필요)
- **좌측:** 기능 플랫 리스트 (그룹핑 없음, 첫 항목 자동 선택)
- **우측 상세 패널:**
  - 최상단: '기능명 + 상태' 한 줄 태그 (큰 글씨)
  - 중단: 구현 항목 체크리스트 (Fact 기반)
  - 하단: 기술 상세 (사용 기술/관련 파일/최근 변경, 가장 작게)
- **⚠️ 확인 필요:** 기획과 다르거나 의도하지 않은 변경 통합. Fix 없이 '→ 검토에서 Fix 확인' 링크만.
- **상태 4종:** ✅구현 완료 / 🔨부분 구현 / 🔲미구현 / ⚠️확인 필요
- **빈 상태 3종:** 기획서+코드 없음 / 기획서만 있고 코드 없음(0% 출발) / 코드만 있고 기획서 없음(Reverse IA)

### 4.3 [검토] — 코드 파손 감지 + Fix (핵심 와우포인트)

사이드바에서 [검토] 선택 시 표시.

- **글로벌 헤더:** 🚨Critical / ⚠️Warning / 💡Info 카운트 + 마지막 분석 시간
- **좌측:** 카테고리 그룹핑 없이 중요도 순 정렬. 각 이슈에 레벨 태그만. Sentry 패턴.
- **우측 상세 패널 — 정보 무게 차등:**
  - 타이틀 (큰 글씨 굵게)
  - → Fact (보통 크기, 1~2문장 비개발자 언어)
  - → Detail (배경 박스, 이슈 유형별 가변)
  - → Fix (강조 배경 박스 + 복사 버튼)
  - → 기술적 근거 (가장 작게)
- **Detail 유형:** 기능 삭제=Before/After 비교(초록/빨간 색상), 기획 불일치=기획 vs 코드 대조, 보안/품질=단순 텍스트, 그 외=단순 텍스트(기본값)
- **Fix 설계 원칙:** Fact 기반 자연어 명령문. '이전에 이랬는데 지금 이렇게 바뀌었으니 되돌려줘' 형태. 창의적 생성이 아닌 사실 기반 복구 지시. 100% 해결 보장이 아닌 '구체적 출발점 제공'.
- **이슈 생명주기:** 미확인 → 확인 완료 → 해결됨(접기/펼치기) / 재감지(최상단 복귀 + 새 Fix 생성)
- **확인 완료 시** 자동으로 CLAUDE.md 가이드라인 등록 (옵트아웃 방식, 유저에게 묻지 않음)
- **Fix 복사 시** '복사되었습니다. CLI에 붙여넣으세요!' 안심 메시지 + 버튼 색상 변경/체크 아이콘

### 4.4 [CLAUDE.md] — AI 가이드라인

- **글로벌 헤더:** '🛡️ AI 가이드라인 N건' + '전체 미리보기' + '전체 복사' 버튼
- **좌측:** '새 가이드라인' / '적용됨' 섹션 분리. 사이드바 배지로 새 가이드라인 수 표시.
- **우측:** 생성된 규칙 텍스트 + 감지 이력 + '이 가이드라인만 복사' + '삭제' 버튼
- **'전체 미리보기' 모달:** 마크다운 형식으로 CLAUDE.md 파일 전체 모습 확인 + 전체 복사
- 유저가 규칙을 작성/수정하지 않음. Asktree가 자동 생성, 유저는 '전체 복사 → 붙여넣기'만.
- MVP: 복사-붙여넣기 방식. Phase 2: MCP 연동 자동 적용.
- 빈 상태: 'CLAUDE.md란?' 자연스러운 교육 안내

### 4.5 [Activity] — 세션별 작업 기록

- **글로벌 헤더:** '전체 세션 N개' + 최근 분석 시간
- **좌측:** 작업명이 헤드라인, 세션 번호+시간은 부제. 최신이 위. 이슈 감지된 세션에 ⚠️ 표시.
- **우측 상세 패널:** AI 번역 요약(1~2문장) + 변경 파일 목록 + '내 프롬프트'(원문) + 감지된 이슈 강조 박스(→ 검토 링크)
- **'원문 보기' 펼치기:** claude-replay 스타일 CLI 로그 전문 렌더링. 삭제=빨간 마스킹, 추가=초록 마스킹. 복사 가능(학습/공유용).
- 추가 토큰 비용 없음: 요약은 검토 분석의 부산물, 원문은 로그 그대로.
- MVP에서는 Should 우선순위. [검토]와 [현황]을 먼저 완성.

### 4.6 [기획서] — 기획 문서 관리

- **좌측:** 업로드된 문서 리스트. 여러 문서 업로드 가능.
- **우측:** 선택된 문서의 추출 기능 목록 + 편집/삭제/추가 가능 + 문서 교체/삭제
- PRD, 기능명세서, 정보구조도 등 어떤 기획 문서든 업로드 가능. 문서 종류 구분 없음.
- AI가 기능 목록 자동 추출. 잘못 추출된 항목은 유저가 직접 수정 가능.
- 권장 형식: .md .txt (.pdf .docx도 지원하지만 텍스트 기반이 가장 정확)
- 버전 관리는 MVP에서 제외. 교체 시 이전 문서 대체.

### 4.7 [설정] — 프로젝트 관리

- 프로젝트 정보: 프로젝트명 수정, 생성일, 사용 도구
- 에이전트 연결: 상태 확인, 마지막 통신, 설치 경로, 재설치/연결 해제
- API 키: Claude API 키 연동 또는 Asktree 크레딧 사용
- 위험 영역: 프로젝트 삭제

### 4.8 사용 흐름

| 단계 | 사용자 행동 | Asktree 동작 |
|------|-------------|--------------|
| ① 회원가입 | asktree.io, Google 로그인 | 계정 생성 |
| ② 프로젝트 생성 | '+새 프로젝트' 클릭 | 프로젝트 생성 → 온보딩 시작 |
| ③ 에이전트 설치 | 터미널에 설치 명령어 붙여넣기 | 백그라운드 서비스 등록. 기존 CLI 로그 초기 동기화. |
| ④ 기획서 업로드(선택) | 기획서 파일 드래그 업로드 | 기능 목록 자동 추출 → 첫 가치 체험 |
| ⑤ 첫 분석 | '분석 시작하기' 클릭 | 코드 유의사항 정적 분석 실행 → 검토 탭으로 이동 |
| ⑥ 코딩 | Claude Code로 평소처럼 작업 | 백그라운드에서 CLI 로그 + 코드 자동 수집. 세션 종료 시 전송. |
| ⑦ 검토 | 이슈 확인 → Fix 복사 → CLI 붙여넣기 | 감지→Fix→CLAUDE.md 가이드라인 자동 생성 |
| ⑧ 가이드라인 적용 | CLAUDE.md 탭에서 전체 복사 → 프로젝트 CLAUDE.md에 붙여넣기 | 다음 세션부터 AI가 가이드라인 참고 |
| ⑨ 현황 확인 | 현황 탭에서 기획 대비 진행률 확인 | PRD vs 코드 대조 결과 표시 |

### 4.9 온보딩 흐름

3-Step 스테퍼. 사이드바 없이 전체 화면 중앙 정렬 집중형 레이아웃.

- **Step 1:** 에이전트 설치 — 설치 명령어 복사 → 터미널 붙여넣기 → 연결 확인
- **Step 2:** 기획서 업로드(선택) — 드래그 앤 드롭 → AI 기능 추출 즉시 표시(첫 가치 체험) → 건너뛰기 가능
- **Step 3:** 첫 분석 — 프로그레스 바 + 분석 단계 체크리스트 → 완료 시 '코드 유의사항 N건 발견' → 검토 탭으로 이동
- MVP: Step 1 에이전트 연결 수동 확인('설치 완료했습니다' 버튼). Phase 2: 자동 감지.

### 4.10 입력 채널

수동 입력 없음. 자동 수집만.

| 채널 | 수집 방식 | MVP |
|------|-----------|-----|
| 프로젝트 파일 구조 | 로컬 에이전트가 자동 스캔 | ✅ |
| CLI 로그 (Claude Code JSONL) | 로컬 에이전트가 ~/.claude/projects/ 자동 감시 | ✅ |
| GitHub 연동 | Phase 2 보조 채널 | Phase 2 |

### 4.11 에러 유형별 감지 범위

| 에러 유형 | 감지 | 대응 |
|-----------|------|------|
| 기존 기능 삭제 (68%) | ✅ | 이상 감지 + Fix + CLAUDE.md 가이드라인 |
| 연쇄 파괴 (20%) | ✅ | 이상 감지 + Fix |
| PRD와 코드 불일치 | ✅ | [현황] 상태 표시 + [검토] Fix |
| 보안 취약 패턴 (OWASP) | ⚠️ 정적만 | 이슈 (Warning 레벨) |
| 미호출 함수 / 가짜 코드 | ✅ | 이슈 (Info 레벨) |
| 실행 순서/런타임 버그 | ❌ 범위 밖 | Sentry 등 모니터링 영역 |

## 5. 기술 아키텍처

### 5.1 설계 원칙

| 원칙 | 설명 |
|------|------|
| Claude Code 전용 (MVP) | JSONL 로그 + CLAUDE.md 가이드라인. Cursor/Aider는 Phase 2. |
| 로컬 에이전트 + 웹 SaaS | 쉘 스크립트 기반 백그라운드 서비스 + asktree.io 대시보드 |
| Smart Push (단방향) | 로컬→서버 HTTP POST. 세션 종료 감지 시 자동 전송. |
| IDE 독립적 구조 | 백그라운드 서비스. IDE/터미널 종료와 무관. |
| Ephemeral Processing | 코드 원본 분석 후 즉시 파기. 분석 결과만 DB 저장. |

### 5.2 기술 스택

| 영역 | 기술 |
|------|------|
| 런타임 | Node.js |
| 웹 프레임워크 | Next.js |
| 서버 DB | Supabase (PostgreSQL) |
| 파일 감시 | chokidar |
| AI 분석 | Claude API |
| 웹 배포 | Vercel |

### 5.3 파일-기능 매핑

AI가 '어떤 코드가 어떤 서비스 기능에 속하는지'를 매핑하는 것이 기술적 핵심 과제.

- 입력 소스 3가지: 기획서 기능 목록, CLI 로그의 유저 프롬프트, 파일명/함수명
- 프롬프트에 "CLI 로그의 유저 명령어를 기반으로 파일-기능 매핑을 수행하라" 지시 포함
- 공용 파일(utils.ts 등)은 기능 매핑 없이 파일 단위로 이슈 처리
- 1주차 프롬프트 품질 검증에서 매핑 정확도 함께 테스트

## 6. UI/UX 상세

### 6.1 정보 무게 차등 원칙

모든 상세 패널에서 정보의 시각적 무게를 차등 적용한다:

- **타이틀/상태:** 가장 크고 굵게 — 0.5초 파악
- **핵심 정보(Fact/구현 항목):** 보통 크기 — 먼저 읽히는 영역
- **근거/상세(Detail):** 배경 박스 안 — '증거'로 제시
- **Fix:** 강조 배경 박스 — '핵심 행동'을 각인
- **기술적 근거:** 가장 작고 연하게 — '궁금하면 보는' 영역

### 6.2 워딩 원칙

- '보호 규칙' → **'AI 가이드라인':** CLAUDE.md 탭 내 용어 통일
- '유저 명령' → **'내 프롬프트':** Activity 탭 내 용어
- '코드 유의사항'은 별도 카테고리 없이 **[검토]의 이슈 레벨 태그(Critical/Warning/Info)로 통합**
- Fact/Detail/Fix 섹션 헤더는 유저에게 **노출하지 않음** (내부 기획 용어). 시각적 무게 차등만으로 구분.
- '기술적 근거'(또는 '기술 상세')만 작은 레이블로 표시

### 6.3 크레딧 소진 시 UX (화면 구상 전)

맛보기 크레딧 10회 소진 시, 분석이 중단되므로 [검토]에 새 이슈가 쌓이지 않음. '크레딧이 소진되었습니다. API 키를 연동하거나 플랜을 업그레이드하세요' 안내 필요. 구체적 화면 위치와 형태는 개발 시 확정.

### 6.4 에이전트 연결 끊김 시 UX (화면 구상 전)

에이전트가 일정 시간 이상 통신 없으면 사이드바 상태 '🔴 미연결'로 변경 + [검토] 탭 상단에 배너 알림 '에이전트 연결이 끊어졌습니다. 재연결이 필요합니다'. 구체적 UI는 개발 시 확정.

## 9. 프롬프트 프로토타이핑 검증 결과

Kamothon(계약서 위험 탐지기, 10개 파일)으로 Claude Sonnet API 테스트 완료.

- **코드 유의사항 감지:** 양호. 6개 실제 문제 감지 (API 키 노출, 미사용 임포트, 빈 모듈 등)
- **PRD 감리:** 매우 우수. 'RAG 시스템이 정의만 있고 호출 안 됨' 정확히 감지.
- **Fix 품질:** Fact 기반 자연어 명령문으로 프롬프트 튜닝 완료. 코드 블록이 아닌 '이전에 이랬는데 지금 이렇게 바뀌었으니 되돌려줘' 형태.
- **AI 품질 전략:** 감지 정확도에 집중, Fix는 '최선의 시도'로 포지셔닝. 100% 해결 보장이 아닌 '구체적 출발점 제공'.
